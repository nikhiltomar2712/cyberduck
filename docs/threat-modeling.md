# Threat Modeling

Threat modeling helps teams think about what can go wrong before incidents happen.

## Simple process

1. Define the system and assets.
2. Draw data flows and trust boundaries.
3. Identify threats.
4. Rank risks by impact and likelihood.
5. Choose mitigations and owners.
6. Revisit after architecture changes.

## STRIDE prompts

- **Spoofing:** can someone pretend to be another user or service?
- **Tampering:** can data be modified without permission?
- **Repudiation:** can actions be denied because logs are weak?
- **Information disclosure:** can private data leak?
- **Denial of service:** can availability be disrupted?
- **Elevation of privilege:** can someone gain more access than intended?
